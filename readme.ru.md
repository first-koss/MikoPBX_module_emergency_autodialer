# MikoPBX Autodialer Module

## Назначение

Модуль предназначен для массового голосового оповещения внешних абонентов через `MikoPBX`.
Запуск выполняется по звонку дежурного на специальный внутренний номер. После запуска модуль
обзванивает всех получателей выбранного сценария, отслеживает результаты и показывает статус
кампании в веб-интерфейсе.

## Зафиксированные требования

### Функциональные

- Основные получатели: внешние абоненты, преимущественно мобильные номера.
- Запуск сценария: по звонку дежурного на заранее определенный внутренний номер.
- После ответа системы дежурный:
  - прослушивает приветственное сообщение;
  - вводит номер сценария;
  - слышит название сценария;
  - подтверждает запуск кодом.
- После запуска система обзванивает всех получателей сценария.
- Остановка кампании в `v1` не требуется.
- Статус кампании отображается на отдельной странице с автообновлением раз в несколько секунд.
- Подтверждение от вызываемого абонента в `v1` не требуется.

### Интеграционные

- Для исходящих вызовов используется `AMI`.
- Для сообщений применяются уже существующие звуковые файлы `MikoPBX`.
- Для переноса между инстансами переносится конфигурация сценариев и списков, а звуковые файлы
  перепривязываются на новом инстансе.

### Правила определения результата

- Успешным оповещением считается случай, когда абонент:
  - ответил на звонок;
  - прослушал сообщение не менее `15` секунд.
- Если абонент ответил, но слушал меньше `15` секунд, попытка считается неуспешной.

### Правила по номерам

- Номер хранится в БД в виде `7XXXXXXXXXX`.
- В интерфейсе номер отображается как `+7(XXX)XXX-XX-XX`.
- CSV-импорт принимает только файлы с заголовками колонок.
- При импорте должна выполняться проверка формата номера.

### Ограничения сценариев

- Номер сценария уникален в пределах точки входа.
- На одну точку входа предполагается не более `10` сценариев.
- Поле `ФИО` хранится как одна строка, без разбиения на части.

### Масштабирование на будущее

- В `v1` точка входа одна.
- Архитектура должна сразу поддерживать будущее расширение на несколько независимых точек входа
  для разных подразделений.

## Целевая архитектура

Модуль строится в двух уровнях:

1. `Scope` или точка входа.
2. Сценарии, списки, кампании и попытки внутри этой точки входа.

## Основные потоки работы

### 1. Запуск сценария по телефону

1. Дежурный звонит на внутренний номер точки входа.
2. Модуль отвечает и проигрывает приветствие.
3. Модуль запрашивает номер сценария.
4. По номеру находится активный сценарий в рамках текущей точки входа.
5. Модуль проигрывает звуковой файл с названием сценария.
6. Модуль запрашивает PIN подтверждения.
7. При успешной проверке создается кампания.
8. На основе привязанных списков формируется снимок получателей кампании.
9. Worker запускает обзвон через `AMI Originate`.

### 2. Исходящий обзвон

1. Worker выбирает получателей кампании порциями с учетом лимита параллельности.
2. Для каждого получателя создается запись попытки дозвона.
3. Через `AMI` инициируется исходящий вызов с передачей идентификаторов кампании и попытки.
4. После ответа абоненту проигрывается звуковое сообщение сценария.
5. Система фиксирует длительность фактического прослушивания сообщения.
6. Если прослушивание составило не менее `15` секунд, попытка считается успешной.
7. Если прослушивание меньше порога или вызов не состоялся, при наличии оставшихся попыток
   получатель переводится в повтор.
8. Кампания завершается, когда все получатели переходят в конечные статусы.

### 3. Мониторинг

- Страница кампании обновляется автоматически каждые `3-5` секунд.
- Пользователь видит:
  - общий прогресс;
  - число успешных оповещений;
  - число неуспешных получателей;
  - статусы отдельных абонентов;
  - историю попыток.

## Технические решения

### Использование существующих возможностей MikoPBX

- Звуковые сообщения сценариев выбираются из уже загруженных звуковых файлов `MikoPBX`.
- Приветствие точки входа и название сценария также выбираются из существующих звуковых файлов.
- Модуль не хранит копии аудиофайлов, а хранит только ссылки на существующие сущности `MikoPBX`.
- Телефонный запуск реализуется поверх стандартной телефонии `MikoPBX`, а не отдельного внешнего сервиса.

### Рекомендуемый способ фиксации 15 секунд прослушивания

Для надежности не стоит определять успешность только по общим `AMI` событиям завершения звонка.
Предпочтительная схема:

- при начале проигрывания сообщения формировать специальный `UserEvent`;
- после достижения порога `15` секунд формировать второй `UserEvent`;
- по второму событию попытка может быть сразу помечена как успешная;
- последующий `Hangup` уже не меняет результат на неуспешный.

Это снизит риск ошибок корреляции и упростит вычисление результата.

### Формат CSV импорта

Поддерживается только CSV с заголовками колонок.

Минимальный обязательный набор колонок:

- `full_name`
- `phone`

Допустимые дополнительные колонки:

- `comment`
- `department`
- `position`
- `external_id`

Правила импорта:

- проверка наличия заголовков;
- проверка обязательных колонок;
- очистка пробелов;
- проверка, что номер соответствует формату `7XXXXXXXXXX`;
- отклонение строк с пустым ФИО или пустым номером;
- формирование отчета по ошибочным строкам;
- недопущение дублирования внутри одного импорта по номеру телефона.

## Этапы реализации

### Этап 1. Рефакторинг шаблона в реальный модуль

  assets и языковые файлы.
- Удалить демонстрационные поля и шаблонные подписи.

### Этап 2. Базовая модель данных

- Создать модели `Scope`, `Scenario`, `RecipientList`, `Recipient`, `ScenarioList`,
  `Campaign`, `CampaignRecipient`, `CallAttempt`.
- Добавить создание и обновление таблиц модуля.
- Подготовить базовую валидацию и связи между сущностями.

### Этап 3. Экран настроек модуля

- Настроить точку входа.
- Настроить PIN подтверждения запуска.
- Настроить приветственное и служебные сообщения.
- Настроить значения по умолчанию для попыток, параллельности и порога успешного прослушивания.

### Этап 4. Экран списков получателей

- CRUD для списков.
- Импорт CSV.
- Экспорт CSV.
- Редактирование состава списка.
- Форматирование телефона в интерфейсе.

### Этап 5. Экран сценариев

- Список сценариев.
- Форма редактирования сценария.
- Привязка файла сообщения.
- Привязка звукового файла названия сценария.
- Привязка одного или нескольких списков получателей.

### Этап 6. Телефонный запуск

- Создать отдельную точку входа по внутреннему номеру.
- Реализовать AGI- или dialplan-сценарий запуска.
- Проверять номер сценария в пределах точки входа.
- Проверять PIN подтверждения.
- Создавать кампанию и снимок получателей.

### Этап 7. Worker обзвона

- Реализовать очередь вызовов.
- Запускать исходящие вызовы через `AMI Originate`.
- Ограничивать число одновременных вызовов.
- Планировать повторные попытки.

### Этап 8. Учет результатов

- Обрабатывать `AMI` события.
- Коррелировать события с конкретной попыткой.
- Фиксировать начало проигрывания и достижение порога успешного прослушивания.
- Пересчитывать статусы получателя и кампании.

### Этап 9. Мониторинг и история

- Список кампаний.
- Карточка кампании.
- Автообновление страницы.
- Просмотр попыток по каждому абоненту.

### Этап 10. Экспорт и импорт конфигурации

- Экспорт сценариев и связей в `JSON`.
- Импорт сценариев на новый инстанс.
- Перепривязка локальных звуковых файлов.
- Проверка отсутствующих зависимостей.

## Критерии готовности v1

Версию `v1` можно считать готовой, если выполнены все условия:

- модуль устанавливается как полноценное расширение `MikoPBX`;
- дежурный может запустить сценарий по внутреннему номеру;
- сценарий выбирается по номеру и подтверждается PIN-кодом;
- модуль обзванивает всех получателей через `AMI`;
- CSV-импорт списков работает только по файлам с заголовками и валидирует номера;
- успешность оповещения определяется по прослушиванию от `15` секунд;
- интерфейс показывает статус кампании с периодическим обновлением;
- сценарии и списки можно переносить между инстансами с перепривязкой аудиофайлов.
